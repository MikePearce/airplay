  <action>
Create a structured Markdown checklist at the path above. The checklist is what the user will run on the Pi, step by step, pasting terminal output back for analysis.

Structure the checklist in strict dependency order (later steps only make sense if earlier ones pass):

**Step 0 — Identify environment:**
- `whoami` (capture the actual username for D-Bus policy references)
- `shairport-sync -V` (capture full version string and compile flags)
- `cat /etc/os-release | head -5` (confirm Pi OS version)

**Step 1 — D-Bus compile flag check:**
- Parse `shairport-sync -V` output for `dbus-interface` in the feature list
- If ABSENT: stop here, document that D-Bus is not compiled in. Include a note that remediation paths (compile from source or accept volume-only) are covered in 01-RESEARCH.md under "Remediation Paths"
- If PRESENT: proceed to Step 2

**Step 2 — D-Bus bus registration:**
- `busctl list | grep -i shairport`
- If empty: check if shairport-sync service is running (`systemctl status shairport-sync`), check if D-Bus is enabled in config (`grep -i dbus /etc/shairport-sync.conf`)
- If registered: proceed to Step 3

**Step 3 — D-Bus policy and access:**
- `ls -la /etc/dbus-1/system.d/ | grep -i shairport` (policy file exists?)
- `cat /etc/dbus-1/system.d/shairport-sync-dbus-policy.conf 2>/dev/null || echo "POLICY FILE MISSING"` (read its contents)
- Check if the policy has `<policy context="default">` with `send_destination` OR a user-specific entry for the username from Step 0
- If policy is missing or restrictive: include the exact XML block to add (from 01-RESEARCH.md, replacing YOUR_USERNAME with actual username) and the reload command (`sudo systemctl daemon-reload && sudo systemctl restart shairport-sync`)

**Step 4 — Live AirPlay playback test (MUST be done while actively streaming):**
- Instruct user to start playing audio from their iPhone/Mac via AirPlay to the Pi
- `dbus-send --system --print-reply --dest=org.gnome.ShairportSync /org/gnome/ShairportSync org.gnome.ShairportSync.RemoteControl.PlayPause`
- Document the three possible outcomes: (a) device pauses = success, (b) AccessDenied = policy issue, go back to Step 3, (c) command succeeds but device does not pause = AirPlay 2 DACP broken
- If outcome (c): `journalctl -u shairport-sync -n 20 --no-pager` to check for DACP activity
- If DACP confirmed broken: note that remediation paths (compile without AirPlay 2 or accept volume-only) are in 01-RESEARCH.md

**Step 5 — MQTT topic flow:**
- `mosquitto_sub -h localhost -t 'shairport-sync/#' -v` (run in one terminal)
- Instruct user to tap a button on the touchscreen (nowplaying.html)
- Expected output: `shairport-sync/remote playpause` (or whichever button was tapped)
- If nothing appears: check `systemctl status mosquitto`, confirm nowplaying.html is connecting to correct broker address and port 9001

**Step 6 — Summary gate:**
- A checklist of the four success criteria from the roadmap, each with a pass/fail checkbox
- Clear statement: "If all four pass, Phase 2 is unblocked. If any fail, paste the failing output and we will determine the remediation path."

For each step, include:
- The exact command to run (copy-pasteable)
- What the expected output looks like
- What a failure looks like and what it means
- Whether to proceed or stop

Do NOT include remediation procedures in the checklist itself — keep it diagnostic. Reference 01-RESEARCH.md for remediation paths.
